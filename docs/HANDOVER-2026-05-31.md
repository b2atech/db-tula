## What Still Needs To Be Done Before Running

### Step 1 — Create `dbtula_app` database

On your QA/app server (NOT prod), using a superuser:

```sql
CREATE DATABASE dbtula_app;
```

### Step 2 — Create `src/B2A.DbTula.Api/appsettings.Development.json`

This file is git-ignored. Create it with your real credentials:

```json
{
  "ConnectionStrings": {
    "AppDb": "Host=YOUR_QA_HOST;Port=5432;Database=dbtula_app;User Id=postgres;Password=YOUR_PASS;"
  },
  "Jwt": {
    "Issuer": "dbtula-api",
    "Audience": "dbtula-web",
    "Secret": "REPLACE_WITH_A_RANDOM_32+_CHAR_STRING_eg_openssl_rand_base64_32"
  },
  "Google": {
    "ClientId": "YOUR_GOOGLE_CLIENT_ID.apps.googleusercontent.com"
  },
  "Cors": {
    "AllowedOrigins": "http://localhost:5173"
  }
}
```

### Step 3 — Set up Google OAuth Client ID

1. Go to [console.cloud.google.com](https://console.cloud.google.com) → APIs & Services → Credentials
2. Create OAuth 2.0 Client ID → Web application
3. Add authorized JavaScript origins:
   - `http://localhost:5173` (dev)
   - `https://your-prod-domain.com` (prod)
4. Copy the Client ID into `appsettings.Development.json` and step 4 below

### Step 4 — Create `web/dbtula-web/.env`

```bash
VITE_GOOGLE_CLIENT_ID=YOUR_GOOGLE_CLIENT_ID.apps.googleusercontent.com
```

### Step 5 — Apply EF Core migration

```powershell
cd C:\Users\bhara\Source\repos\db-tula
dotnet ef database update --project src/B2A.DbTula.Api/B2A.DbTula.Api.csproj
```

This creates all 4 tables in `dbtula_app`.

### Step 6 — Register databases in the UI

Once running, sign in as the first user (auto-Admin), go to `/databases`, and register:
- Your QA databases (read-only credentials) — `isWriteAccount = false`
- Your PROD databases (read-only credentials) — `isWriteAccount = false`
- For sync apply: Create a write Postgres role on PROD with additive-only permissions:

```sql
-- On prod DB, run as superuser:
CREATE ROLE dbtula_writer LOGIN PASSWORD 'a_strong_password';
GRANT CONNECT ON DATABASE your_db TO dbtula_writer;
GRANT USAGE ON SCHEMA public TO dbtula_writer;
GRANT CREATE ON SCHEMA public TO dbtula_writer;  -- allows CREATE TABLE, CREATE INDEX, etc.
-- Do NOT grant DROP or TRUNCATE
```

Then register this write account in the UI with `isWriteAccount = true`, linking it to the read account.

## How the Comparison Flow Works

1. User selects source + target DB on `/compare`, clicks "Run"
2. API creates `ComparisonRun` (Status=Pending), enqueues run ID to `Channel<Guid>`
3. `ComparisonWorker` (background service) picks up the run ID
4. Decrypts credentials, creates schema providers, calls `SchemaComparer.CompareAsync()` (same code as CLI)
5. Sends SignalR progress events → React shows live log
6. Generates safe/risky/destructive sync scripts, stores in `ComparisonRun` table
7. React shows results table with filters, diff modal for mismatches

## How Apply-Safe Works

1. Admin clicks "Apply Safe Changes" on a completed run
2. UI shows confirmation modal
3. `POST /api/comparisons/{id}/apply-safe` — Admin only
4. API finds the write account for the target DB (must be registered separately)
5. Decrypts write account credentials, connects to prod
6. Executes SAFE section statements one-by-one (try/catch per statement)
7. Inserts `SyncApplyLog` row with full SQL + success/failure counts
8. Returns results to UI

**Important:** RISKY and DESTRUCTIVE sections cannot be applied via UI — download only.

## Security Architecture

- **Google ID token** validated server-side via `Google.Apis.Auth.GoogleJsonWebSignature.ValidateAsync()` — client secret never goes to browser
- **JWT** stored in `httpOnly; Secure; SameSite=Strict` cookie — not accessible to JavaScript
- **Connection strings** encrypted at rest with ASP.NET Core Data Protection (AES-256 CBC + HMAC) — never returned to frontend
- **Write credentials** stored separately from read credentials — registered as `isWriteAccount=true`
- **Postgres write role** should have no `DROP` privilege at the database level — enforced at DB, not just UI
- **Audit log** is immutable — every apply action recorded with who, when, what SQL, how many succeeded/failed
